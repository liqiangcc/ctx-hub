# Context Hub 需求文档

## 1. 项目简介

Context Hub 是一个面向个人和 AI 的上下文集中管理工具。

它用于集中记录和查询日常工作中的高价值信息，例如服务地址、接口文档、常用命令、构建方式、排查经验、历史问题、他人同步的信息和临时注意事项。

用户和 AI 可以通过关键词、标签或唯一 Key 快速找到需要的信息。

一句话定义：

> Context Hub 是用户和 AI 共用的外部记忆库。

---

## 2. 背景

日常工作中的信息通常分散在：

- 聊天记录
- 浏览器收藏
- Wiki 和文档平台
- 代码仓库
- 命令历史
- 个人笔记
- 大脑记忆

这些信息经常包括：

- 某个服务的地址在哪里
- 接口文档在哪里
- 日志应该怎么看
- 某个模块怎么构建
- 某个问题以前怎么排查
- 某条命令怎么写
- 他人同步过哪些注意事项

当前存在的问题：

- 信息分散，查找耗时
- 常用地址、命令和规则容易忘记
- 有价值的同步信息容易被聊天记录淹没
- AI 缺少项目上下文，只能猜测
- 同一类信息被反复询问、反复整理
- 不同 AI 工具无法复用同一份上下文

因此需要一个集中、可查询、可复用的上下文管理工具。

---

## 3. 建设目标

系统应实现：

1. 将有价值的信息集中存储。
2. 支持低成本快速录入。
3. 支持关键词全文查询。
4. 支持标签筛选。
5. 支持通过唯一 Key 精确定位。
6. 支持快速查看和复制查询结果。
7. 支持人和 AI 查询同一份上下文。
8. 减少人的记忆负担和重复查找成本。
9. 减少 AI 对地址、命令、构建方式和项目规则的猜测。
10. 为后续 MCP 接入提供统一上下文入口。

核心原则：

```text
先集中，能搜索，再逐步结构化。
```

---

## 4. 使用对象

### 4.1 个人用户

个人用户负责记录和查询信息。

典型操作：

- 记录一个接口文档地址
- 记录一条常用命令
- 记录一个服务的构建方式
- 记录一次故障排查经验
- 记录他人同步的重要信息

### 4.2 AI

AI 可以查询 Context Hub 获取上下文，再继续完成分析或操作。

AI 可以使用的信息包括：

- 服务信息
- 文档地址
- 常用命令
- 构建规则
- 历史排查经验
- 临时注意事项

AI 不应直接猜测这些信息。

---

## 5. 典型使用场景

### 5.1 查询服务信息

用户或 AI 查询：

```text
order-service
```

期望找到：

- Git 仓库地址
- 服务负责人
- 测试环境地址
- 接口文档地址
- 日志查询说明
- 构建命令
- 相关排查经验

### 5.2 查询接口文档

查询：

```text
payment api
支付接口文档
```

期望快速找到 Swagger、Apifox、YApi 或其他接口文档地址。

### 5.3 查询常用命令

查询：

```text
order-service 构建
查看支付日志
启动本地服务
```

期望找到可复制的命令。

### 5.4 查询构建上下文

查询：

```text
order-service JDK
release 1.8 Maven
```

期望找到：

- JDK 版本
- Maven 版本
- settings 配置说明
- profile
- 标准构建命令

### 5.5 查询排查经验

查询：

```text
支付失败
401
mock token
```

期望找到之前记录的排查步骤和历史结论。

### 5.6 查询同步信息

用户将聊天、会议或他人同步的信息记录后，可以通过服务名、关键词或标签重新找到。

### 5.7 AI 获取上下文

AI 在处理问题前先查询：

- 服务信息
- 构建规则
- 文档地址
- 常用命令
- 历史排查经验

再决定是否查询日志、代码、数据库或执行构建。

### 5.8 快速复制

搜索到地址、命令或正文后，可以直接复制指定内容用于：

- 浏览器访问
- 终端执行
- 粘贴给 AI
- 编写文档
- 故障排查

---

## 6. 信息类型

首期支持记录以下信息：

- 普通笔记
- 服务信息
- 环境信息
- 文档链接
- 常用命令
- 构建上下文
- 排查手册
- 历史问题
- 临时通知
- 同步信息
- 数据库和表说明
- 测试规则
- 服务依赖说明

首期不强制所有信息严格分类。

---

## 7. 数据字段需求

### 7.1 必需字段

每条记录至少包含：

- 标题
- 内容
- 创建时间

### 7.2 推荐字段

建议支持：

- 唯一 Key
- 标签
- 信息类型
- 关联服务
- 关联环境
- 来源
- 更新时间
- 失效时间
- 可信度
- 状态

示例：

```yaml
key: runbook.payment.failed
title: 支付失败排查规则
tags:
  - payment
  - test
  - runbook
service: payment-service
env: test
type: runbook
source: 支付组同步
created_at: 2026-06-21
expires_at:
content: 支付失败时先查询 payment_callback_log，再查询 payment-service 日志。
```

---

## 8. 核心功能需求

### 8.1 快速新增

支持快速记录：

- 一段文本
- 一个地址
- 一条命令
- 一段聊天内容
- 一条排查经验

录入时不应强制填写大量字段。

### 8.2 关键词查询

支持对以下字段进行查询：

- 标题
- 正文
- Key
- 标签
- 服务
- 环境
- 来源

支持中文、英文、数字、错误码、服务名和命令内容。

### 8.3 标签查询

支持通过一个或多个标签筛选，例如：

```text
payment
maven
test
command
runbook
```

一条记录可以拥有多个标签。

### 8.4 Key 精确查询

重要信息可以设置唯一 Key，例如：

```text
service.order-service.info
build.order-service.jdk17
command.maven.compile
runbook.payment.failed
doc.payment.api
```

用户和 AI 可以通过 Key 精确获取记录。

### 8.5 结果展示

查询结果至少展示：

- 编号
- Key
- 标题
- 标签
- 匹配摘要
- 来源
- 创建或更新时间

### 8.6 查看详情

支持通过搜索结果编号或 Key 查看完整记录。

### 8.7 快速复制

支持复制：

- 整条记录
- 正文
- URL
- 命令
- Key
- 标题

### 8.8 标签列表

支持查看已有标签及其对应记录数量，减少标签重复和命名混乱。

### 8.9 信息更新

支持修改已有记录，并保留更新时间。

### 8.10 信息失效管理

支持标记：

- 临时信息
- 已过期信息
- 已废弃信息

查询结果应提示信息是否可能过期。

---

## 9. AI 使用需求

AI 应能够：

- 按关键词查询上下文。
- 按标签筛选上下文。
- 通过 Key 获取完整记录。
- 获取某个服务的相关信息。
- 获取相关命令和构建规则。
- 获取历史排查经验。

AI 应遵守：

```text
不知道服务地址、构建方式、文档位置或常用命令时，优先查询上下文管理工具，不直接猜测。
```

AI 默认不应：

- 删除记录。
- 修改关键配置。
- 读取敏感凭证。
- 自动执行记录中的命令。

---

## 10. 安全约束

不得直接保存或返回：

- 密码
- Token
- Cookie
- SSH 私钥
- 数据库密码
- API Secret
- 生产密钥

允许保存凭证引用：

```yaml
credential_ref: vault://order/test-readonly
```

系统应支持对疑似敏感信息进行提示或限制。

---

## 11. 范围边界

Context Hub 负责：

- 信息记录
- 集中存储
- 关键词查询
- 标签筛选
- Key 定位
- 内容展示
- 快速复制
- 为 AI 提供上下文

首期不负责：

- 自动执行命令
- 连接服务器
- 查询日志文件
- 查询数据库
- 构建代码
- 管理 Kubernetes
- 自动同步聊天软件
- 自动抓取网页
- 自动判断信息真伪
- 自动解决业务问题

这些能力应由其他独立工具或 MCP 提供。

---

## 12. 首期范围

首期必须支持：

1. 新增一条文本记录。
2. 设置标题和标签。
3. 可选设置唯一 Key。
4. 关键词查询。
5. 标签查询。
6. Key 精确查询。
7. 查看完整记录。
8. 快速复制内容。
9. 修改和删除个人记录。
10. 支持 AI 通过 MCP 只读查询。
11. 防止返回明文敏感信息。

首期不要求：

- 向量检索
- 语义检索
- 自动分类
- Web 管理后台
- 多人权限系统
- 聊天软件自动同步
- 复杂知识图谱

---

## 13. 验收标准

首期满足以下条件即可使用：

- 用户能够快速录入信息。
- 用户可以通过关键词找到已记录内容。
- 用户可以通过标签缩小查询范围。
- 用户可以通过 Key 精确找到一条记录。
- 查询结果可以快速复制。
- AI 可以查询同一份上下文。
- AI 无法获取未授权的敏感信息。
- 无查询结果和查询失败能够明确区分。
- 信息规模增加后仍可正常查询。
- 上下文库可以作为用户和 AI 的统一信息入口。

---

## 14. 核心定义

> Context Hub 是用户和 AI 共用的外部记忆库。它负责将零散信息集中存储，并通过关键词、标签和 Key 快速找回，让用户少记、少找，让 AI 少猜、少错。
